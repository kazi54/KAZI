How it works

01

Create a tenant config

Declare the client's org ID, preferences, secrets, and destination adapters in a single YAML file.

02

Connect their destinations

Notion, Salesforce, HubSpot, Airtable, webhook, email — whatever CRM they use. KAZI OS routes to it.

03

Run scoped pipelines

Same workflow, different tenant context. Outputs are isolated, logs are scoped, secrets never cross boundaries.

Tenants are just config files

Adding a client is adding a YAML file. No code changes. No redeployment. No database migrations.

  • Org ID scopes all pipeline runs and outputs
  • Secrets resolve from environment variables at runtime
  • Destinations map to named adapters (review, publish, archive)
  • Preferences customize behavior per client
T tenants/silicon-xchange.yaml
org_id: silicon-xchange
display_name: Silicon Xchange
domain: content

destinations:
review:
adapter: notion
config:
workspace: ${SX_NOTION_WORKSPACE}
database: ${SX_NOTION_DB}
publish:
adapter: webhook
config:
url: ${SX_PUBLISH_WEBHOOK}

preferences:
timezone: Africa/Nairobi
output_format: markdown

CRM-agnostic delivery

KAZI OS does not know what a CRM is. It knows what a destination is. You write adapters once, tenants declare which ones they use.

  • Built-in adapters: Notion, Webhook, Email
  • Community adapters: Salesforce, HubSpot, Airtable
  • Custom adapters: extend BaseDestination in ~30 lines
  • Destinations are named keys — pipelines reference keys, not platforms
D adapters/salesforce.py
from kazi import BaseDestination

class SalesforceDestination(BaseDestination):
name = "salesforce"

async def push(self, payload, context):
# Create record in Salesforce
record = await self.client.create(
object_type=self.config["object_type"],
data=payload
)
return DestinationReceipt(
id=record.id,
status="delivered"
)

async def check_status(self, receipt_id):
return await self.client.get_status(receipt_id)

Run across all tenants at once

One command runs the same pipeline for every configured tenant. Each run is isolated — scoped logs, scoped outputs, scoped secrets.

  • Run all tenants: kazi run produce --all-tenants
  • Run one tenant: kazi run produce --tenant acme
  • Dry-run: kazi run produce --dry-run
  • Scoped logs per tenant for debugging

Built for multi-tenancy

Secret Isolation

Tenant secrets resolve from environment variables at runtime. Never stored in config. Never cross boundaries.

Destination Routing

Each tenant declares their own destinations. Same pipeline, different delivery targets. Zero code changes.

Scoped Logs

Every pipeline run is tagged with org_id. Filter, query, and debug per tenant without noise.

Per-Tenant Scheduling

Different tenants, different schedules. One runs daily at 6am, another weekly on Mondays. Same pipeline.

Adapter Registry

Built-in adapters for Notion, Webhook, Email. Community adapters for Salesforce, HubSpot. Write your own in 30 lines.

Usage Tracking

Per-tenant run counts, compute time, and output volume. Know what each client costs and produces.
